# Роадмап изменений технологического ландшафта компании "Будущее 2.0"

## Обзор

Роадмап отражает поэтапные изменения в технологическом ландшафте компании с привязкой к бизнес-целям. Каждый этап
включает конкретные проекты, ожидаемые результаты, ответственные команды и необходимые ресурсы.

## Ключевые бизнес-цели

1. **Быстрая подготовка отчётности** - Генерация управленческих отчётов за секунды/минуты
2. **Независимое развитие направлений** - Финтех и ИИ-направления развиваются независимо
3. **Масштабирование для новых бизнесов** - Быстрое подключение партнёров (фармацевтика, электроника)
4. ** Качество медицинских и финансовых сервисов** - Надёжность, безопасность, соответствие требованиям

---

## Этап 1: Фундамент (0-3 месяца)

### Бизнес-цель

Решение критичных проблем производительности и создание основы для дальнейшего развития.

### Проекты

#### Проект 1.1: Миграция в облачную инфраструктуру

**Цель**: Развёртывание облачной инфраструктуры для масштабирования и гибкости.

**Задачи**:

- Выбор облачного провайдера (AWS/Azure/GCP)
- Настройка базовой инфраструктуры (VPC, сети, безопасность)
- Развёртывание Kubernetes кластера
- Настройка CI/CD пайплайнов
- Миграция первых сервисов в облако

**Ожидаемые результаты**:

- Облачная инфраструктура готова к использованию
- Kubernetes кластер развёрнут и настроен
- CI/CD пайплайны работают для облачных сервисов
- Первые сервисы мигрированы в облако

**Ответственные команды**:

- **DevOps команда** (ведущая)
- **Архитекторы** (консультации)
- **Команды доменов** (миграция сервисов)

**Необходимые ресурсы**:

- 2-3 DevOps инженера
- 1 Архитектор
- Облачная инфраструктура (бюджет: ~$10-20K/месяц на старте)
- Инструменты: Terraform, Kubernetes, CI/CD платформа

**Связь с бизнес-целями**:

- **Масштабирование**: Основа для быстрого добавления новых бизнесов
- **Качество сервисов**: Геораспределённое развёртывание для отказоустойчивости

**Задачи**:

- Настройка DWH и подключение к источникам данных
- Миграция критичных данных из SQL Server
- Настройка ETL процессов для загрузки данных
- Тестирование производительности

**Ожидаемые результаты**:

- Облачный DWH развёрнут и настроен
- Критичные данные мигрированы из SQL Server
- ETL процессы загружают данные ежедневно
- Производительность запросов улучшена в 5-10 раз

**Ответственные команды**:

- **Data Engineering команда** (ведущая)
- **Аналитики** (тестирование запросов)
- **DevOps команда** (инфраструктура)

**Необходимые ресурсы**:

- 2-3 Дата-инженера
- 1-2 Аналитика
- Облачный DWH (бюджет: ~$5-15K/месяц)
- Инструменты: dbt, Airflow, SQL клиенты

**Связь с бизнес-целями**:

- **Быстрая отчётность**: Разделение compute и storage, автоматическое масштабирование
- **Масштабирование**: Поддержка сотен терабайт с линейным масштабированием

**Ожидаемые результаты**:

- Критичные отчёты генерируются за секунды вместо часов
- Витрины обновляются автоматически (ежедневно/еженедельно)
- Пользователи получают доступ к быстрым отчётам через Power BI

**Ответственные команды**:

- **Data Engineering команда** (ведущая)
- **Бизнес-аналитики** (определение требований)
- **Пользователи** (тестирование отчётов)

**Необходимые ресурсы**:

- 2-3 Дата-инженера
- 1-2 Бизнес-аналитика
- Время пользователей для тестирования
- Инструменты: dbt, SQL, Power BI

**Связь с бизнес-целями**:

- **Быстрая отчётность**: КРИТИЧНО - решение проблемы медленной генерации отчётов
- **Влияние на выручку**: Быстрые отчёты = быстрые решения = больше прибыли

## Этап 2: Доменная архитектура (3-6 месяцев)

### Бизнес-цель

Обеспечение независимого развития финтех- и ИИ-направлений без изменений в DWH.

### Проекты

#### Проект 2.1: Выделение доменов и создание собственных хранилищ

**Цель**: Создание независимых доменов с собственными хранилищами данных.

**Задачи**:

- Финальное определение границ доменов (Медицинский, Финтех, ИИ, Аналитический, Интеграционный)
- Создание собственных хранилищ для каждого домена (PostgreSQL, MongoDB, Object Storage)
- Миграция операционных данных из DWH в домены
- Настройка репликации и бэкапов для доменных хранилищ
- Тестирование изоляции доменов

**Ожидаемые результаты**:

- 5 доменов созданы с собственными хранилищами
- Операционные данные мигрированы из DWH в домены
- DWH используется только для аналитики
- Домены изолированы и независимы

**Ответственные команды**:

- **Архитекторы** (проектирование доменов)
- **Команды доменов** (миграция данных)
- **Команда дата-инженеров** (миграция данных)
- **DevOps команда** (инфраструктура хранилищ)

**Необходимые ресурсы**:

- 2 Архитектора
- 5-7 Разработчиков (по 1-2 на домен)
- 2-3 Дата-инженера
- 2 DevOps инженера
- Хранилища данных (PostgreSQL, MongoDB, Object Storage)

**Связь с бизнес-целями**:

- **Независимое развитие**: КРИТИЧНО - домены могут развиваться независимо
- **Масштабирование**: Легко добавлять новые домены для партнёров

**Задачи**:

- Настройка Schema Registry для версионирования событий
- Определение событий для взаимодействия доменов
- Реализация event producers и consumers в доменах
- Миграция интеграций с Apache Camel на Kafka
- Настройка мониторинга событий

**Ожидаемые результаты**:

- Kafka кластер развёрнут и настроен
- Основные события между доменами определены и реализованы
- Домены взаимодействуют через события (асинхронно)
- Мониторинг событий работает
- Apache Camel постепенно выводится из эксплуатации

**Ответственные команды**:

- **Integration команда** (ведущая)
- **Команды доменов** (реализация producers/consumers)
- **DevOps команда** (инфраструктура Kafka)

**Необходимые ресурсы**:

- 2-3 Инженера интеграций
- 3-5 Разработчиков из доменов
- 1-2 DevOps инженера
- Kafka кластер (бюджет: ~$3-8K/месяц)

**Связь с бизнес-целями**:

- **Независимое развитие**: КРИТИЧНО - слабая связанность через события
- **Масштабирование**: Легко добавлять новых потребителей событий

- Настройка rate limiting и мониторинга

**Ожидаемые результаты**:

- Аутентификация и авторизация работают

**Ответственные команды**:

- **Команда интеграций** (ведущая)
- **Команда IT безопаснотси** (требования безопасности)

- 2-3 Разработчика из доменов
- 1 DevOps инженер
- API Gateway (Kong / AWS API Gateway)

**Связь с бизнес-целями**:

- **Независимое развитие**: Единая точка интеграции, но независимая разработка
-  **Качество сервисов**: Централизованная безопасность и мониторинг
- **Масштабирование**: Легко добавлять новые API для партнёров

- 2-3 Разработчика

**Задачи**:

**Ответственные команды**:
